# ForkPlay — Master Build Plan
### Git for Gamers: open-source, AI-built, community-governed games

---

## 1. The Vision

ForkPlay is a platform where AI-built, open-source browser games are published, played, forked, remixed, and governed by their communities.

Three pillars, no platform combines today:

1. **GitHub-style forking** — every game is a git repo; anyone can fork, branch, and propose changes. Every branch of every game is playable in one click.
2. **OSRS-style governance** — communities vote on proposals (70% pass threshold). Democracy with an exit door: if your idea loses the vote, fork it and build it anyway.
3. **A component marketplace as the standard** — games are assembled from interoperable parts (entities, behaviors, systems, assets). The marketplace enforces the technical standard through incentive, not rules: compliance unlocks a free library of parts.

**Scope for v1:** single-player browser games only. Multiplayer is explicitly out of scope until the platform is proven.

---

## 2. Locked Architecture Decisions

These decisions are final for v1. Every phase prompt assumes them. Do not relitigate mid-build.

| Decision | Choice | Why |
|---|---|---|
| Git backend | GitHub API (Octokit), each game = one repo | Forking/branching/merging for free; migrate to self-hosted Gitea later if needed |
| Game runtime | Browser-only, served in sandboxed iframe with strict CSP | One-click play of any branch; security from day one |
| Engine standard | Custom lightweight SDK ("ForkPlay SDK") on TypeScript + Canvas/WebGL | Required for behavior-level interoperability across games |
| Two game tiers | **Ecosystem games** (built on SDK, full marketplace access) and **Open games** (any web tech, fork-and-play only) | Standard via incentive; door stays open for everything else |
| Structure enforcement | Scaffold template repo + publish-time validator + the platform's webhook-triggered build pipeline. **The platform pipeline IS the CI** — do not also create GitHub Actions on game repos | The cheap path must be the compliant path; one validation system, not two |
| Tunables | All balance values live in `config.json`, never in code | Most community votes become one-line JSON diffs |
| Default license | MIT for code, CC-BY for assets, enforced at upload | Prevents legal mess; forks stay legal |
| Platform stack | Next.js (App Router) + TypeScript + Tailwind + Postgres/Prisma + GitHub OAuth | Boring, fast, well-known to AI models |
| Votes | Proposals with 70% approval threshold; owner retains veto in v1 | Simple defaults; forking is the escape valve |
| SDK versioning | Games pin an exact SDK version in `game.json` (`sdkVersion`); SDK follows semver; builds install the pinned version | A v1.1 SDK release must never silently change or break published games |
| SDK distribution | SDK published to **public npm** as `@forkplay/sdk` (validator CLI included). The build worker installs the pinned version from npm; the monorepo uses workspace linking only for internal development | Standalone game repos and user forks build outside the monorepo — the SDK must be installable from a public registry or no real game can build |
| Part versioning | Catalog parts are versioned; games reference `partId@version`; updates are opt-in per game | A part update must never silently change every game using it |
| Save data | SDK storage API namespaces all persistence by `gameSlug + branch` automatically; raw localStorage use fails validation **(ecosystem tier only — open-tier games may use any storage and accept the cross-branch save risk)** | Switching branches or playing a fork must never corrupt saves in ecosystem games |
| Artifact storage | Built game artifacts in S3-compatible object storage (S3/R2), served via CDN from a separate origin | Cheap, scalable, and isolates untrusted game code from the platform origin |
| Deployment topology | Next.js app on Vercel (or container); build pipeline runs on a **separate worker** (Docker on a VPS/Fly/ECS) consuming the job queue | Serverless cannot run sandboxed multi-minute untrusted builds |
| Art & audio (v1) | Procedural/geometric placeholder assets generated by code, one coherent visual style; a post-launch "asset quality pass" replaces them | AI coding agents produce poor pixel art; honest placeholders beat bad art |
| Open-tier validation | Open games get a lighter gate: valid manifest + license + builds to static /dist + loads without console errors. No SDK/structure checks | Keeps the door open without diluting the ecosystem standard |
| Fork naming | Fork slug = `{original-slug}--{username}` (auto, editable later); display name defaults to "Original Name (username's fork)" | One-shot prompts trip on unstated conventions; collisions prevented by design |
| Game storage isolation | Game iframes get `sandbox="allow-scripts"` ONLY (opaque origin — strongest isolation). Games therefore CANNOT use localStorage/indexedDB directly; the SDK storage API is a **postMessage bridge**: the parent platform page persists data namespaced by `gameSlug + branch`. SDK ships a dev-shim (in-memory + JSON file) so games work standalone in `npm run dev` | `allow-scripts` alone makes browser storage throw SecurityError; adding `allow-same-origin` on a shared artifact origin would let every untrusted game read every other game's saves. The bridge resolves both and enables future cloud saves |
| Library distribution | Library published to **public npm** as `@forkplay/library`; games pin an exact `libraryVersion` in `game.json`; `partId@version` resolves within that release. User-published marketplace parts are **vendored into the game repo** at remix time (under `src/vendored-parts/`) | Standalone repos and the build worker must resolve parts from a public registry; vendoring keeps user parts fork-safe without a private registry |
| Artifact serving | A dedicated **artifact server** (built in 4A, owns storage access) serves `/artifacts/{game}/{branch}/{path}` by streaming from the bucket with correct content-types, the strict game CSP, and immutable cache headers. Port 3001 locally; the separate artifact domain in prod. **Never presigned URLs** (they break relative asset paths) and never raw MinIO/S3 (they can't set CSP) | The iframe needs index.html + relative assets with exact headers; this is load-bearing security and an ambiguity a one-shot would improvise badly |
| Repo visibility (v1) | **Public repos only**, enforced at publish time. Anonymous clone in the worker — no worker GitHub token needed | Open-source games are the product thesis; private-repo support is a paid-tier question for later |
| Webhooks | The **GitHub App owns the webhook**: one app-level webhook (push events) covers every installed repo automatically. No per-repo hook creation, no `admin:repo_hook` OAuth scope ever | Per-repo webhook registration was an unowned step that fell between phases; app-level delivery closes it structurally |
| Governance commit credential | Auto-commits from passed proposals use a **GitHub App installation** on the game repo, never the owner's stored OAuth token. App ID + private key live in env; **installation IDs are per-Game database rows**, captured during 4B's ecosystem publish flow (which prompts the owner to install the app) | OAuth tokens expire and get revoked; a Phase 7 auto-commit failing silently months later is unacceptable. Installations are per-repo facts, not config |

## 3. Build Order Logic

The dependency chain — why phases are in this order:

```
Infrastructure (Phase 0 — accounts, storage, environments)
  → SDK (the contract)
    → Component Library 2A: behaviors + systems (the logic)
      → Component Library 2B: entities, art, audio, UI, FX
        → Seed Games (proof the parts compose)
          → Build Worker 4A (repo → validated artifact, in isolation)
            → Platform Site 4B (a place to play them)
              → Fork Engine (play any branch)
              → Marketplace (browse and remix the parts)
                → Governance (votes that become diffs)
                  → Launch hardening
```

Each phase is one single-shot prompt. Each prompt ends at a **handoff point**: a precise set of artifacts and interfaces the next prompt plugs into. Run each prompt in a fresh Claude Code session inside the monorepo, point it at this file **plus ENVIRONMENT.md (the machine contract: pre-installed tooling, the no-sudo rule, and the BLOCKED.md escalation protocol)** and the prior phase's handoff artifacts, and let it run.

**Monorepo layout (created in Phase 1, used by all phases):**

```
forkplay/
├── packages/
│   ├── sdk/            # Phase 1
│   └── library/        # Phases 2A + 2B
├── games/              # Phase 3 (seed games; later moved to own repos)
├── platform/           # Phase 4+ (Next.js site)
├── infra/              # Phase 0 (topology docs, env templates)
├── templates/
│   └── game-scaffold/  # Phase 1
├── DECISIONS.md        # every phase appends its assumptions here
└── MASTER-PLAN.md      # this file
```

**Frozen ≠ immutable — the patch-release protocol.** "Frozen" means the schema shape and public API contracts cannot change. It does NOT forbid bug fixes. When a later phase finds a bug in a frozen, published package (Phase 3 will find SDK bugs — plan on it): the session fixes the bug in the monorepo **without changing any contract**, bumps the PATCH version (0.1.0 → 0.1.1), runs the pack dry-run, and writes a `[CRITICAL]`-style entry titled `[PUBLISH]` to BLOCKED.md — the human publishes at the next gate, and the session repins affected games in the same change. Contract-breaking fixes are different: those halt for a human decision, full stop. This keeps freezes honest without making bugs unfixable.

# PHASE 0 — Infrastructure & Accounts (Human Setup, ~1 hour)

**Goal:** everything later prompts assume exists. This phase is mostly *you*, not the AI — accounts and credentials a one-shot prompt cannot create. Skipping this is the #1 way later phases stall.

### ☑ Manual checklist (do these yourself)
1. **Run `setup-kali.sh` once** (the one-time machine prep: system packages, Docker, nvm/Node 22, gh auth, local Postgres + MinIO). Re-login afterward for the docker group. Then place **ENVIRONMENT.md** in the repo root — it is the machine contract every AI session reads first.
2. **GitHub org** created (e.g. `forkplay-games`) — seed games and forks live here.
3. **GitHub OAuth App** registered (client ID/secret) for platform login + repo operations.
4. **Postgres**: local dev DB already running via Docker from step 1; provision managed Postgres (Neon/Supabase/RDS) only when deploying.
5. **Object storage**: local MinIO already running from step 1; create the real S3/R2 bucket only when deploying.
6. **Worker host** chosen for the build pipeline (Fly.io, a VPS with Docker, or ECS). Decide now; Phase 4A builds against it (locally, the worker runs as a Docker container on this machine).
7. **Domains**: one for the platform app, one *separate* domain/subdomain for serving game artifacts (security isolation — locked decision). Needed at deploy time, not for local dev.
8. **npm account** created (and `npm login` done) — you publish `@forkplay/sdk` (Phase 1 gate) and `@forkplay/library` (Phase 2B gate) manually; the org/scope `@forkplay` must be claimed now.
8b. **GitHub App** registered under the org with: contents read/write permission, **push-event webhook enabled at the app level** (per the locked Webhooks decision), webhook secret set. Generate the private key; key + secret go in `.env`.
8c. **smee.io channel** created (one click at smee.io) and set as the App's webhook URL for local dev; the channel URL goes in `.env` as WEBHOOK_PROXY_URL. Swap to the real endpoint at deploy time.
9. **Anthropic API key / Claude Code** ready for the build sessions.

# PHASE 1 — The SDK: Schema + Runtime Core

**Goal:** define the contract everything else obeys. An entity-component runtime where games are compositions of entities, behaviors, and systems, with all tunables in data files.

**Why first:** the SDK is the standard. Library, games, marketplace, and governance all depend on its schema. Getting this wrong is the only unrecoverable mistake in the plan.

### ▶ SINGLE-SHOT PROMPT — PHASE 1

```
You are building Phase 1 of ForkPlay (read MASTER-PLAN.md in repo root
for full context; obey all Locked Architecture Decisions).

Build the ForkPlay SDK: a lightweight TypeScript entity-component game
runtime for 2D browser games, in packages/sdk/.

DELIVERABLES

1. SCHEMA (packages/sdk/src/schema/) — Zod schemas + exported TS types:
   - game.json manifest: { name, slug, description, version,
     engine: "forkplay-sdk", sdkVersion (exact semver pin),
     libraryVersion (exact semver pin; REQUIRED for ecosystem
     tier, omitted for open tier), entryPoint, license, authors[],
     tier: "ecosystem" | "open" }
   - entity definition: { id, sprite, size, position, behaviors[],
     tags[], layer }
   - behavior definition: { id, type, params (typed per behavior type) }
   - system definition: { id, type, params }
   - config.json: flat key/value store of ALL tunable numbers
     (speeds, costs, spawn rates, health values). Behaviors and
     systems reference tunables by key, e.g. "$cfg.playerSpeed" —
     never hardcode balance numbers.
   - scene definition: { id, entities[], systems[], tilemap?,
     background, music? }

2. RUNTIME (packages/sdk/src/runtime/):
   - Game loop (fixed timestep update, rAF render) on HTML5 Canvas
   - Entity manager: spawn/destroy, tag queries
   - Behavior engine: behaviors are pure-ish functions
     (entity, world, params, dt) => mutations; attachable/detachable
     at runtime
   - Built-in primitives needed to prove the model (full library
     comes in Phase 2): transform/velocity, AABB collision +
     collision events, input mapping (keyboard + touch),
     sprite renderer (static + sheet animation), audio player,
     scene loader that builds a running game from JSON definitions
   - Storage API per the Game-storage-isolation locked decision:
     a postMessage BRIDGE. In production the game runs in an
     opaque-origin iframe (sandbox="allow-scripts" only) where
     localStorage THROWS — the SDK storage API posts save/load
     messages to the parent page, which persists namespaced by
     gameSlug + branch. Define the message protocol in the SDK
     now — versioned, and validated by SOURCE IDENTITY, not origin
     strings: opaque-origin iframes report event.origin === "null"
     and replies must use targetOrigin "*", so the parent matches
     event.source === iframe.contentWindow and both sides exchange
     a per-session nonce in a handshake. Ship a dev-shim (in-memory +
     JSON file) so `npm run dev` works standalone. The validator
     must FAIL any ECOSYSTEM-tier game touching raw
     localStorage/indexedDB (open tier is exempt) — branch/fork
     switching must never corrupt saves.
   - Audio player must NO-OP cleanly when Web Audio is absent
     (jsdom/headless) — smoke tests run without an AudioContext.

3. SCAFFOLD (templates/game-scaffold/):
   - game.json, config.json, /src/scenes/, /src/custom-behaviors/,
     /assets/, /tests/smoke.test.ts (game boots headless, runs 60
     simulated frames, no errors), README explaining the structure
   - npm scripts: dev, build (outputs static /dist), test

4. VALIDATOR (packages/sdk/src/validate/):
   - CLI: `forkplay validate <dir>` — checks manifest, schema
     compliance, the storage rule, and the MECHANICAL no-magic-
     numbers rule: a numeric literal in behavior/system params
     FAILS unless its key is on the structural whitelist defined
     in the schema (position, size, layer, zIndex, frame indices,
     and similar non-balance keys — define the whitelist
     explicitly). Everything else must be a $cfg reference. Also
     verify every partId@version reference resolves within the
     CATALOG of the pinned libraryVersion — a game must not pin
     library 0.1.0 while referencing a part version that only
     exists in 0.2.0. Then runs the smoke test.
     Exit 0 = publishable.

5. PROOF: examples/pong/ — minimal Pong built ONLY from JSON
   definitions + SDK primitives. If Pong needs custom code, the
   primitives are too weak; fix the SDK, not the example.

CONSTRAINTS
- Zero runtime dependencies beyond the platform (Zod for schema ok)
- Every public API has TSDoc comments
- Vitest for tests; meaningful coverage on runtime core
- Package as @forkplay/sdk ready for public npm publish (proper
  exports map, files whitelist, semver 0.1.0, validator exposed as
  a bin). Do a dry-run pack; the human runs `npm publish` at the
  gate (publishing is a core-path human action, not yours)
- Append all assumptions made to DECISIONS.md

Ask me nothing about requirements — make sensible assumptions and
append them to DECISIONS.md. For BLOCKERS, follow ENVIRONMENT.md's
two-tier escalation: core-path blockers (DB, pipeline, storage,
auth, SDK contract) HALT the session with a [CRITICAL] entry;
peripheral blockers are logged and routed around.
```

### ✅ Definition of Done
- `forkplay validate examples/pong` passes
- Pong is playable via `npm run dev`
- All schemas exported as both Zod validators and TS types
- `npm pack` dry-run produces a clean publishable @forkplay/sdk tarball; human publishes v0.1.0 to npm at this gate

# PHASE 2B — Component Library: Entities, Art, Audio, UI, FX

# PHASE 4A — The Build Worker

**Goal:** the riskiest, most foundational service in the platform, built and proven in isolation: a standalone worker that turns (repoUrl, branch, commit) into a validated, stored, servable artifact. No web app yet — if this works, Phase 4B is conventional CRUD; merged together, a pipeline bug hides inside a half-built website.

### ▶ SINGLE-SHOT PROMPT — PHASE 4A

```
You are building Phase 4A of ForkPlay (read MASTER-PLAN.md,
DECISIONS.md, ENVIRONMENT.md). Build the build worker as a
standalone service in platform/worker/ — no web UI in this phase.

THE WORKER
A long-running Node service (runs in Docker locally per the
deployment topology decision) that consumes build jobs from a
Postgres-backed queue table and executes:
  clone (shallow, ANONYMOUS — v1 is public repos only per the
  locked decision; visibility is enforced at publish, not here) →
  detect tier from game.json →
  install deps including the PINNED sdkVersion from npm →

  console errors, using system Chromium per ENVIRONMENT.md) →
  build /dist → upload artifact to S3-compatible storage
  (MinIO locally; real S3/R2 via the same env-configured client) →
  write Build row (status, logs, artifactPath).

REQUIREMENTS
1. Sandboxing: each build in an ephemeral SIBLING container — the
   worker mounts the host Docker socket and launches builds beside
   itself (NOT Docker-in-Docker; see ENVIRONMENT.md). Builds use a
   dedicated BUILDER IMAGE (define its Dockerfile: Node 22 +
   Chromium + build deps — the host's Chromium is not visible
   inside containers) in an explicit two-stage flow:
   Stage 1 WITH network: clone + npm install. Stage 2 with
   `--network none`: validate + build, then the worker uploads
   from the extracted artifact. CPU/memory/time limits from env;
   workspace destroyed after upload.
   CONTAINER PLUMBING (do not improvise these — see ENVIRONMENT.md):
   a. Networking: from inside the worker container, localhost is
      NOT the host — Postgres/MinIO are loopback-bound on the host.
      Locally, attach the worker to the infra compose network and
      address services by name (db:5432, minio:9000) via env;
      `--network host` is the acceptable fallback.
   b. Workspaces: the Docker daemon resolves -v paths ON THE HOST —
      the worker's internal filesystem is invisible to sibling
      build containers. Use NAMED Docker volumes for build
      workspaces shared between worker and builder (create per
      build, destroy after).
   c. The builder image (Node 22 + Chromium + build deps) will be
      1.5–2 GB — note it in DECISIONS.md per the disk-hygiene rule;
      this is expected, not a blocker.
2. Database: create the Prisma schema for Build and the job queue

   4B can add tables without migrating these).
3. CLI harness: `worker build <repoUrl> <branch>` triggers one
   job end-to-end from the terminal — this is how the phase is
   tested without any web app.
4. Failure quality: validator errors and build logs stored
   verbatim and readable; a failed build must explain itself.
5. Throughput: process N jobs concurrently (env-configured),
   per-(game,branch) deduplication so rapid pushes don't stack
   redundant builds.
6. THE ARTIFACT SERVER (per the locked Artifact-serving decision):
   a small companion service in platform/artifact-server/ serving
   GET /artifacts/{game}/{branch}/{path} — streams from the bucket,
   sets correct content-types, the strict game CSP, and immutable
   cache headers. Port from ARTIFACT_SERVER_PORT. No presigned
   URLs; no direct bucket exposure. Include a header-assertion
   test (fetch index.html and one asset; assert CSP + content-type).

PROOF: run the CLI against all six seed repos in
games/PUBLISHED.md (clean clones, npm SDK — no workspace links)
and one deliberately broken repo. Six green artifacts in MinIO,
one readable rejection.

append them to DECISIONS.md. For BLOCKERS, follow ENVIRONMENT.md's
```

## 5. Known Risks & Future Forks in the Road (post-v1)

- **Votes-to-code gap for feature requests** — config and part-swap proposals auto-apply; free-text features still need a human or AI contributor. Future: platform-hosted AI agent that drafts the PR for passed feature proposals.
- **Asset quality pass** — v1 ships procedural placeholder art and synthesized audio by design. Post-launch: commission or AI-generate a real asset pack as new catalog part versions; games upgrade opt-in via the part-versioning system.
- **SDK upgrade tooling** — version pinning protects games but will fragment the ecosystem across SDK versions over time. Future: a "test my game on SDK vX" one-click branch build + migration codemods.
- **Fork divergence** — shared SDK + config-as-data keeps most forks mergeable, but deep custom-behavior forks will drift. Acceptable for v1; revisit with "sync from parent" tooling.
- **GitHub dependency** — fine for v1; the repo wrapper is intentionally thin so a Gitea/Forgejo migration stays feasible.
- **3D and richer engines** — the schema's `engine` field and the open-tier leave the door open; do not attempt before the 2D ecosystem is alive.
- **Multiplayer** — explicitly after this entire plan ships and the single-player loop is proven.
