# AUDIT — games/breakout

**Auditor pass:** instrumented headless simulation (real `createGame` + `createLibraryRegistry`, hundreds of frames, simulated input) **+** real-browser play (Chrome-for-Testing via puppeteer-core against the built `dist/`) **+** comparison to intended behavior (README + MASTER-PLAN "what breakout must prove"). No code changed; harness lived in `/tmp` and is not committed.

## Verdict

**PLAYABLE — but DEGRADED on two of its intended points.**

The core loop is real and correct: the ball launches, bounces off the walls and paddle, breaks bricks, a lost ball costs a life, lives run out → game-over (lose), and clearing the wall → game-over (win). Scoring, HUD, title/pause/game-over chrome, and the mobile pad all work; the game renders cleanly in a real browser with zero real console errors. It is winnable (a controlled paddle cleared all 50 bricks → `outcome=win`, score 2500).

Two intended behaviors are **not** delivered as advertised:
1. **Ball reflection off bricks is wrong for side/horizontal hits** — the ball tunnels straight through entire brick rows instead of bouncing (root cause in the frozen SDK `reflect-on-hit`).
2. **There is no "next level"** — clearing the wall ends the game with a *win*; no second brick layout is ever loaded. The advertised "level-progression to the next level" is a counter increment that immediately satisfies the win condition.

## Probed and found FINE (no defect)

- **contact-damage one-tick seed delay** — benign for breakout. Bricks' `health-and-death` seeds `state.hp` from frame 1 (bricks are static and present at scene start), long before the ball reaches them, so `contact-damage`'s "skip un-seeded victim" guard (`contact-damage.ts:45`) never delays a real brick hit. Bricks break reliably on contact.
- **lives-respawn double-listener / "play again" double-counting** — **none.** `lives-respawn` polls `world.state` (no event listener) and stores scratch via `systemState`, which `Game.loadScene` clears (`game.ts:112-123`). The host `GameShell` registers its `gameover`/`screenFx` listeners **once** in its constructor, not per run. RUN 7: after `loadScene("main")`, `world.state` fully reset, lives reseeded to 3, no instant decrement, no double game-over. The DECISIONS "loadScene doesn't reset world.events" hazard does not bite this game.
- **Lives + lose game-over** — RUN 1 (idle paddle): lives drained 3→2→1→0, then `gameOver`, `outcome=lose`. Respawn after delay works.
- **Win game-over** — RUN 6 (controlled paddle): wall fully cleared → `outcome=win`, score 2500 (= 50 bricks × `blockScore` 50), 1 life lost.
- **Ball–paddle reflection** — bounces upward correctly, no sticking; `vy` magnitude is preserved/grown (never collapses), so there is **no shallow-angle death / stuck ball**.
- **Scoring + HUD** — `+50`/brick into `world.state.score`; HUD score and lives text bind and update live; rendered correctly in-browser.
- **Title / pause / game-over chrome + mobile pad** — all render (browser screenshots): BREAKOUT title card, the full 50-brick rainbow wall, score (left) + lives (right) HUD, working pause overlay (Resume / Quit to title) that freezes the sim, and 2 touch buttons present in the DOM. **Zero real console errors** — the only 404s were `/favicon.ico` (browser auto-probe; production artifact-server answers favicon with 204 per DECISIONS).

## Prioritized fix list

### Game-local fixes (Bucket A)
1. **F2 (Major):** Decide what "level progression" means for breakout. Either (a) be honest and reframe it as a single-screen "clear the wall to win" (drop the `level`/`winLevel` framing, win directly on `clearTag`), or (b) deliver real levels — add additional brick-layout scenes and host glue that loads the next layout on the `level-up` event (and raise `winLevel`). As shipped it neither progresses nor matches MASTER-PLAN's "to the next level."
2. **F5 (Polish):** Reorder the paddle behaviors so `clamp-to-world` runs **after** `velocity` (`move-4dir → velocity → clamp-to-world`) to kill the one-frame wall overshoot.
3. **F6 (Polish):** Optionally implement a serve-from-paddle (ball parented to paddle until a launch key) and/or set the respawn position relative to the paddle, with some launch-angle variety.

### Library-patch candidates (Bucket B — FROZEN packages; flag for a [PUBLISH] triage session, do not fix here)
1. **F1 (Major) — `@gitcade/sdk` `reflect-on-hit`:** add an `axis:"auto"` (minimum-translation) mode using the existing `overlapAxis()` so the correct velocity component flips per collision side. Without it, accurate block/wall reflection is impossible from data alone. Blast radius today: breakout; structurally: every future brick/pinball/wall game.
2. **F3 (Minor) — `@gitcade/sdk` `reflect-on-hit`:** `maxSpeed` clamps only the reflected axis while `english` adds to the other axis uncapped → total speed exceeds the cap. Clamp total (or other-axis) speed, or rename/document `maxSpeed` as per-axis. Blast radius: breakout + the pong example.
3. **F4 (Minor, informational):** `contact-damage` damages all overlapping targets while `reflect-on-hit` reacts to one — by design, but worth noting for any game that packs damageable targets tightly.

## Coverage / what I could NOT fully exercise
- **Touch controls** were verified to exist (2 buttons in the DOM) and the synth-key path is straightforward host glue, but I did not drive them via real pointer taps in the browser.
- **High-score persistence across sessions** through the production storage bridge was not exercised — standalone uses the in-memory shim; I confirmed `score`/`highScore` increment in `world.state`, but the cross-session bridge round-trip is Phase 4B parent-side territory, out of this game's scope.
- I confirmed the **win** outcome via headless simulation (RUN 6) rather than watching a full clear in the browser; the game-over card uses the same shell overlay as the (verified) pause card.
